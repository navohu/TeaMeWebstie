Use Cases

Use Case 1

Use Case: Choose a team

ID: UC1

Brief Description:

The users are able to choose a team

Primary Actors:

Users

Secondary Actors:

None

Preconditions:

Users must be logged in to the system

Main Flow:

  1. Use case starts when user selects “Choose a team”
  2. The system directs the user to a new page displaying the set of teams the user is a member of
  3. User chooses one the teams and clicks on its name
  4. The system takes the user to the feelings page

Postconditions:

The user is in the team he/she has selected

Alternative flows:

If the user is not a member of any team, the system displays nothing instead of list of the teams.



Use Case 2

Use Case: Create a new team

ID: UC2

Brief Description:

The users can create a new team

Primary Actors:

Users

Secondary Actors:

Users added to the team

Preconditions:

Users must be in the user page

Main Flow:

  1. Use case starts when user selects “Choose a team”
  2. The system directs the user to a new page displaying the set of teams the user is a member of
  3. User selects “+” in the up right corner of the page
  4. The system shows a new page to create a new team
  5. User types a team name and clicks create
  6. The system creates a new team
  7. User types the usernames of the users he/she wants to add and clicks add
  8. The system adds all the users the current user enters

Postconditions:

The system will add a new team and make the current users admin of that new team

Alternative flows:

If the user is not a member of any team, the system displays nothing instead of list of the teams.



Use Case 3

Use Case: Edit user details

ID: UC3

Brief Description:

The users can update their personal details

Primary Actors:

Users

Secondary Actors:

None

Preconditions:

Users must be logged in and in the user page

Main Flow:

  1. Use case starts when user selects to edit their profile
  2. The system takes the user to the pages of their personal details
  3. User makes some changes to the current details
  4. User clicks submit
  5. The system takes the user to the user page

Postconditions:

The system will update and save all the details the user has edited and saved

Alternative flows:

If the user clicks back button without saving changes, the system will direct the user to user page without updating the profile details




Use Case 4

Use Case: Send a message in the TeaRoom

ID: UC4

Brief Description:

The users can have a chat with all the people in the TeaRoom

Primary Actors:

Users

Secondary Actors:

The other users in the TeaRoom

Preconditions:

Users must be logged in to the system

Main Flow:

  1. Use case starts when user selects to go to the TeaRoom
  2. The system takes the user to the TeaRoom
  3. User selects the message box and types their messages
  4. User send their messages
  5. The system displays the messages on the screen

Postconditions:

The system will display the messages for all the users who are currently in the TeaRoom

Alternative flows:

If there are no users in the TeaRoom, the messages of the current use will not be seen by others



Use Case 5

Use Case: Post in the news feed

ID: UC5

Brief Description:

The users are able to write posts in the news feed

Primary Actors:

Users

Secondary Actors:

The other users

Preconditions:

Users must be logged in to the system

Main Flow:

  1. Use case starts when user chooses a team and goes to the home page of that team
  2. The system takes the user to the news feed page
  3. User selects the are separated for writing posts
  4. Users writes their posts and shares
  5. The system displays the post on the screen

Postconditions:

The system will display the messages for all the members of the team in which the current user have written a post

Alternative flows:

None
